# COMPARISON.md — treeLine's generated /newest POM vs the hand-written one

I built [treeLine](https://github.com/jamesmyers4/treeLine) before this assignment existed: an open-source engine that crawls a site, captures DOM and accessibility-tree snapshots, runs them through tiered AI interpretation, and generates Playwright page objects, specs, and selector stability rankings — all gated behind human review. That is the same architecture QA Wolf's platform runs. So I pointed it at Hacker News (one polite crawl, 5 pages, read-only) and did the thing the architecture demands: reviewed the AI's proposal against the page object I wrote by hand for this assignment. AI proposed, human reviewed and overrode where the human knew better.

The generated inputs are in this directory: [the POM and specs](generated-pom-newest.md), [the selector stability ranking](selector-stability-newest.md), [the testid audit](testid-audit.md), [the site atlas](site-atlas.md), and [the axe scan](axe-report.md). The hand-written side is `pages/HNListPage.ts`, `pages/HNNewestPage.ts`, and `pages/StoryRow.ts`.

## Where the tool and the hand agree

- **Reject positional selectors on this site.** treeLine marked all 261 absolute XPaths and 231 of 261 CSS paths for /newest unstable, and top-ranked role-based selectors (229 of them, every one rated stable). The hand-written suite made the same call from the other direction: auto-waiting locators anchored to HN's few semantic hooks, never to table positions. Two independent processes — one statistical, one judgment — landing on the same locator philosophy is real validation.
- **The premise of the assignment's difficulty.** The testid audit reads 0% coverage across 1,121 interactive elements on 5 pages. That one number is why locator strategy on HN is a skill demonstration at all: there are no test attributes to lean on, only structure and semantics.
- **What the page is.** The atlas identified /newest as "the newest submitted stories … in reverse chronological order" with story title, submission time, rank, and item ID as its key entities — independently confirming that the entities my suite scrapes and asserts on are the ones that define the page.
- **The structural landmarks.** Where treeLine's CSS paths traverse story content, they pass through exactly the anchors the hand-written POM uses: `tr.athing.submission`, `td.title > span.titleline > a`, `span.sitebit.comhead`, `td.subtext > span.subline`, `a.hnuser`, `span.age`. The tool saw the same skeleton I did.

## Where the hand-written POM wins, and why the tool couldn't get there

**The two-row bridge.** HN renders each story as two sibling `<tr>` elements — a `tr.athing` title row and an anonymous subtext row — related by nothing but adjacency. `StoryRow` bridges them with `xpath=following-sibling::tr[1]`, so one object exposes title, rank, author, age, and comments together. treeLine recorded both rows' selectors faithfully and never connected them: its POM has the title link and that story's age link as unrelated fields, ~15 entries apart. This isn't a selector-ranking problem — no amount of ranking connects two elements. It requires comprehending that two DOM structures are one logical entity, which is exactly the kind of cross-element insight current generation pipelines don't attempt.

**The timestamp source.** treeLine's proposal for a story's age is `getByRole("link", { name: "3 minutes ago" }).nth(0)` — the visible relative text, which is recomputed at render time and useless for a sort assertion. The hand-written POM reads the `title` attribute on the parent `.age` span, which carries the raw ISO-plus-unix timestamp (`"2026-07-22T07:14:03 1784704443"`). Choosing the machine-readable attribute over the human-readable text is a data-quality judgment about what an assertion needs, not a selector preference — and it's invisible to a ranker that only scores whether a selector resolves uniquely and survives recrawls.

**`.age`, not `.age a`.** Same locator family, one level apart: the timestamp attribute lives on the span, not the anchor inside it. treeLine targeted the anchor (it's the interactive element — correct for clicking, wrong for reading). I originally made the identical mistake and caught it from Playwright's failure snapshot in June. The difference is that a human carries the correction forward; a generator remakes the choice fresh every crawl.

**One class instead of 260 fields.** The generated POM names every element of the captured snapshot: `sEAMVAHybridDecoupledRISCVVectorProcessorLink`, `jimmc414Link3`, `hideLink12`. The hand-written equivalent is one `StoryRow` class instantiated per row, with `getStories(count)` producing all thirty. The structural fact the generator missed is that /newest is thirty instances of one repeating pattern — and on a page whose content fully turns over in minutes, locators bound to specific story titles were dead before the crawl finished writing them.

## An honest limitation, stated plainly

treeLine models the page it captured, not the template that produced it. On a live feed that distinction is fatal to the generated POM as-is: every content-derived name and every `.nth()` disambiguation index encodes the story order of one instant on 2026-07-22, so the artifact is stale before a human can review it. The selector _rankings_ and the _audits_ survive — they describe the structure — but the generated page object itself is only a starting scaffold, and pretending otherwise would be selling the tool past what it does. Deriving the repeating-row abstraction from repeated structure is the obvious next capability, and it's not built yet.

## Cross-check: two independent axe scans

treeLine's crawl ran its own axe-core pass, fully independent of this suite's Session 5 baseline (different machine role, different axe integration, different day's stories). Agreement on /newest: **7 of 7 rule ids identical** — `color-contrast`, `image-alt`, `label`, `landmark-one-main`, `link-name`, `page-has-heading-one`, `region` — with zero rules found by one scanner and not the other. Node counts differ (treeLine reports 273 raw color-contrast nodes; the baseline normalizes to 22 structural findings by design), which is the expected delta between a raw scan and a churn-normalized baseline. Two independent scanners converging on the identical violation set is strong evidence the committed `a11y-baseline.json` describes the site, not the tooling.

## Feedback filed against treeLine

Bugs and improvements this exercise surfaced, going back into the treeLine issue tracker:

1. **Generated POMs can fail to compile.** Elements named "3 minutes ago" become `readonly 3MinutesAgoLink1: Locator` — a leading digit is not a legal TypeScript identifier, so `newest.page.ts` is a syntax error as emitted. Fix: prefix-sanitize identifiers, and add a `tsc --noEmit` gate on every generated artifact before it reaches human review — a proposal that doesn't parse wastes the reviewer's time budget.
2. **Stability ranking trusts entity-id selectors.** The 30 CSS selectors rated "stable" on /newest are all `#up_<storyid>` vote anchors. The id is unique and survives a recrawl of the same snapshot, but it embeds a story id — it's the least stable selector on the page across time. The ranker should treat selectors containing entity-shaped tokens (long digit runs, per-item ids) as unstable by construction, the same normalization insight this suite's a11y baseline needed.
3. **Detect repeating regions and emit a row component.** Thirty sibling `tr.athing` rows with identical internal structure should generate one `StoryRow`-style class plus an indexed accessor, not ~230 per-instance fields. This is the single highest-leverage change for a QA engineer consuming the output: it turns a stale snapshot inventory into a reusable abstraction.
4. **Proposed specs should verify their actions against the captured DOM.** The proposed /newest spec clicks `getByRole("button", { name: /submit|create|save|continue|send/i })`, but HN's search form has no submit button — it submits on Enter. The element wasn't in the snapshot treeLine itself captured, so the proposal generator could have known. The skip-wrap and the honest "unverified guess" comments are the right design; emitting actions pre-checked against the snapshot would make the review gate faster to clear.
5. **Surface machine-readable attributes in reports.** The `.age` span's `title` attribute carries the exact timestamp a test wants to assert on, and no report mentions it. An "assertable data sources" section — elements with `title`, `datetime`, or `data-*` attributes carrying machine-readable values — would hand a test author the single most valuable fact about this page.
6. **Entity extraction is inconsistent across pages.** The atlas abstracts /newest's entities cleanly (Story title, Points, Submission time, Item ID) but lists thirty individual story titles as "key data entities" for / and /front. Same site, same crawl, different abstraction level — the /newest treatment should be the norm.
7. **Deduplicate the selector report.** 410 KB for 5 pages, mostly thirty near-identical row entries per page. Reporting each repeating pattern once with an instance count would cut it an order of magnitude and make the stable/unstable signal readable at a glance.

## Why this document is in a take-home

QA Wolf's model is AI generation gated by human review: the platform proposes, and an engineer who knows the product decides what ships. treeLine is my independently built version of the same bet, down to the skip-wrapped proposed specs waiting on a reviewer. This comparison is that review, performed in public against my own tooling: where the AI's proposal validated the hand-written suite, I said so; where the human knew better, I overrode it and wrote down why; where my own tool fell short, that's filed as feedback above rather than smoothed over. That loop — propose, review, override, feed back — is the job.
